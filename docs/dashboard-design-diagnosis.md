# Dashboard Design Diagnosis

작성일: 2026-03-31  
대상: Claude Pulse Web Dashboard  
범위: 디자인 진단 및 와이어프레임 이전 단계의 구조 평가  
원칙: 소스 수정 없이 현재 구현과 스펙을 기준으로 분석

## 1. 검토 대상

- 구현 레이아웃: `src/web/src/App.tsx`
- 디자인 토큰: `src/web/src/index.css`
- 패널 구현:
  - `src/web/src/components/ActivityStream.tsx`
  - `src/web/src/components/CostEstimate.tsx`
  - `src/web/src/components/ErrorPanel.tsx`
  - `src/web/src/components/SessionTimeline.tsx`

  - `src/web/src/components/AgentTracker.tsx`

- 의도 스펙: `docs/dashboard-panels.md`

## 2. 한 줄 진단

현재 대시보드는 "정리된 미니멀 UI"로는 성립하지만, "실시간 운영 대시보드"로서의 정보 위계와 상태 전달력은 아직 부족하다.

## 3. 핵심 결론

1. 레이아웃의 큰 방향은 맞다.
2. 그러나 카드 간 우선순위가 약해 중요한 상태가 먼저 보이지 않는다.
3. 일부 패널은 이름과 실제 시각화 방식이 다르다.
4. 빈 상태 화면의 완성도는 높지 않고, 데이터가 들어와도 운영 관점의 빠른 판단을 충분히 돕지 못한다.
5. 다음 단계는 "예쁘게 다듬기"보다 "무엇을 먼저 보게 할 것인가"를 재설계하는 것이 우선이다.

## 4. 현재 구조 진단

### 4.1 레이아웃 구조

현재 메인 화면은 다음 구조를 가진다.

- 상단 헤더
- 1행: `Activity` 대형 패널 + `Cost`, `Errors` 세로 스택
- 2행: `Timeline`, `Files`, `Agents`
- 3행: `Server`, `Projects`

근거:

- `src/web/src/App.tsx:27-77` 헤더
- `src/web/src/App.tsx:81-106` 12-column 그리드

평가:

- 장점: 가장 자주 보게 되는 `Activity`를 크게 준 점은 적절하다.
- 장점: 우측에 `Cost`, `Errors`를 배치한 것은 보조 KPI 패널 구조로 자연스럽다.
- 한계: 2행과 3행 카드들이 모두 거의 동일한 시각적 무게를 가져, 운영 우선순위가 평평해진다.
- 한계: 카드 높이가 대부분 고정되어 있어 데이터량이 적을 때는 공허하고, 많을 때는 카드 내부 스크롤이 늘어난다.

### 4.2 헤더 구조

헤더는 브랜드명, 세션 경과 시간, 툴 호출 수, 에러 수, 세션 ID 일부, Live/Offline 상태, 테마 토글로 구성되어 있다.

근거:

- `src/web/src/App.tsx:29-65`

평가:

- 장점: 헤더를 얇게 유지해 본문 집중도를 방해하지 않는다.
- 한계: 상태 바라기보다 브랜드 바에 가깝다.
- 한계: 현재 상태가 "정상", "주의", "위험" 중 어디인지 즉시 해석하기 어렵다.
- 한계: 활성 에이전트 수, 최근 에러 발생 시점, 서버 상태 요약 같은 운영형 핵심 지표가 빠져 있다.

진단:

헤더는 지금보다 더 명확한 상황판 역할을 해야 한다. 현재 정보는 존재하지만 행동 유도력이 낮다.

## 5. 시각 시스템 진단

### 5.1 색상 토큰

근거:

- `src/web/src/index.css:3-26`

평가:

- 장점: 다크/라이트 테마 토큰 분리가 되어 있고, 의미 색상도 제한적으로 사용한다.
- 장점: `accent`, `red`, `green`의 역할이 비교적 명확하다.
- 한계: `surface`, `border`, `text-faint`가 서로 너무 가까워서 카드 구획과 텍스트 계층이 약하다.
- 한계: 빈 상태 화면에서 화면 전체가 동일 밀도의 어두운 톤으로 눌려 보인다.

진단:

미니멀함은 유지되지만, "읽기 쉬운 다크 대시보드"라기보다 "톤 다운된 카드 모음"에 가깝다.

### 5.2 타이포그래피

근거:

- `src/web/src/index.css:29-33`
- `src/web/src/App.tsx:29`
- 각 카드 제목의 `text-[13px]` 사용

평가:

- 장점: 숫자에 `tabular-nums`를 적용해 로그/수치 UI의 정렬감을 챙겼다.
- 한계: 제목, 설명, 보조 수치, 비어 있음 상태가 대부분 작은 폰트로 수렴해 위계가 약하다.
- 한계: 카드 제목이 전반적으로 너무 작고 얇아 스캔 포인트가 되지 못한다.

### 5.3 카드 시스템

근거:

- `src/web/src/index.css:47-52`

평가:

- 장점: 일관된 `card` 토큰으로 전체 화면이 통일된다.
- 한계: 거의 모든 카드가 동일한 반경, 동일한 테두리, 동일한 패딩, 동일한 표면 명도를 가져 중요도 차이를 시각적으로 설명하지 못한다.

진단:

카드 시스템은 안정적이지만, 대시보드에서 필요한 "강조 카드", "경고 카드", "보조 카드"의 계층이 없다.

## 6. 패널별 진단

### 6.1 Activity Stream

근거:

- `src/web/src/components/ActivityStream.tsx:18-24`
- `src/web/src/components/ActivityStream.tsx:39-64`
- 스펙: `docs/dashboard-panels.md` 1번 섹션

현재 구현:

- `tool-start`, `tool-error`만 추려 표 형식으로 나열
- 시간, 도구명, 경로 또는 명령 문자열 표시
- 에러인 경우 메시지 추가 노출

장점:

- 로그형 이벤트를 빠르게 스캔할 수 있다.
- 도구별 색상 분리는 실용적이다.
- 실시간 스크롤 이동은 라이브 피드 감각을 준다.

문제:

- 스펙에 있는 액션 상세, 필터링, 에이전트 단위 맥락이 없다.
- 이벤트가 단순 행 단위라 "작업 흐름"보다 "로그 줄"처럼 보인다.
- 에러 행이 일반 행과 동일 구조라 중요도가 충분히 튀지 않는다.
- 파일 경로와 명령이 한 칸에 섞여 있어 정보 타입이 불명확하다.

디자인 판단:

이 패널은 현재도 기능은 하지만, 운영 대시보드의 메인 패널로 보기에는 스토리텔링과 계층이 약하다.

### 6.2 Cost Estimate

근거:

- `src/web/src/components/CostEstimate.tsx:27-61`
- 스펙: `docs/dashboard-panels.md` 3번 섹션

장점:

- 핵심 수치와 범위를 한 카드 안에 간결하게 보여준다.
- 도구별 분포를 작은 막대로 압축한 방식은 공간 효율이 좋다.

문제:

- 주요 수치가 크지만, 왜 이 숫자가 중요한지 설명 문맥이 약하다.
- 분포 막대가 모두 동일 색이라 정보 구분력이 낮다.
- 실제 스펙의 "세션 비용 인사이트"보다는 단순 비용 숫자 카드에 가깝다.

디자인 판단:

현재 형태는 KPI 카드로는 적절하지만, 대시보드 내 우선순위는 `Errors`보다 낮게 보이면서도 시선은 더 먼저 가져간다.

### 6.3 Error Panel

근거:

- `src/web/src/components/ErrorPanel.tsx:7-48`
- 스펙: `docs/dashboard-panels.md` 7번 섹션

장점:

- 카드 내 에러 블록이 별도 배경과 좌측 경고선으로 구분된다.
- `stderr` 일부를 함께 보여주는 점은 실용적이다.

문제:

- 실제 스펙은 도구 실패, 서버 에러, 테스트 경고를 통합해야 하지만 구현은 `tool-error`만 표시한다.
- 빈 상태 문구가 `Clean` 하나뿐이라 정보성이 너무 약하다.
- 가장 중요한 패널이어야 할 수 있는데, 화면에서 강한 경고 영역으로 보이지 않는다.

디자인 판단:

기능과 표현 모두 아직 "알림 센터" 수준까지 올라오지 못했다. 현재는 도구 에러 로그 카드에 가깝다.

### 6.4 Session Timeline

근거:

- `src/web/src/components/SessionTimeline.tsx:15-63`
- 스펙: `docs/dashboard-panels.md` 4번 섹션

장점:

- 타임라인 패턴 자체는 세션 회고용 컴포넌트로 적절하다.
- 점과 세로선 구조는 이해하기 쉽다.

문제:

- 스펙은 세션 전체를 이야기 구조로 보여주지만, 구현은 단순 이벤트 축약 리스트다.
- 이벤트 라벨이 지나치게 단순해 "session start", "agent done" 수준에서 멈춘다.
- 파일 경로도 축약 표시만 되어 사건의 의미가 충분히 드러나지 않는다.

디자인 판단:

이름은 `Timeline`이지만 실제로는 `Recent Events`에 더 가깝다.

### 6.5 File Heatmap

근거:

- `src/web/src/components/FileHeatmap.tsx:16-57`
- 스펙: `docs/dashboard-panels.md` 5번 섹션

장점:

- 상위 파일 접근 빈도를 빠르게 보는 용도로는 충분히 쓸 수 있다.
- 1위 항목을 `accent`로 강조한 점은 명확하다.

문제:

- 스펙상 "트리맵/히트맵"인데 실제 구현은 순위형 막대 리스트다.
- 읽기/수정/쓰기 성격이 합산되어 파일의 성격 차이가 사라진다.
- 경로 축약이 2-depth만 남아 정보 충돌 가능성이 있다.

디자인 판단:

이 패널은 이름과 결과물이 가장 크게 어긋난다. 지금 표현을 유지할 거면 명칭 변경이 필요하다.

### 6.6 Agent Tracker

근거:

- `src/web/src/components/AgentTracker.tsx:14-72`
- 스펙: `docs/dashboard-panels.md` 6번 섹션

장점:

- 현재 실행 중 에이전트에만 녹색 강조를 준 점은 적절하다.
- 경과 시간 노출은 실시간 감각을 준다.

문제:

- 스펙의 활동량 정보가 없다. 현재는 타입과 경과 시간만 보인다.
- `running/done` 외에 무슨 작업을 하는지 알 수 없다.
- 상태 리스트 이상으로 발전하지 못했다.

디자인 판단:

이름은 Tracker지만 실제로는 `Agent Status List` 수준이다.

### 6.7 Server Monitor

근거:

- `src/web/src/components/ServerMonitor.tsx:35-77`
- 스펙: `docs/dashboard-panels.md` 2번 섹션

장점:

- 로그 뷰어처럼 읽기 쉬운 모노스페이스 표현을 사용했다.
- `ERR`, `WRN` 강조는 기본 기능을 한다.

문제:

- 스펙에 있는 서버 이름, 포트, 상태, 요청 수, 평균 응답시간 같은 운영 지표가 없다.
- 전체가 로그 테이블 중심이라 모니터보다 콘솔 뷰어에 가깝다.
- 정상 상황과 위험 상황의 차이를 상단에서 즉시 요약해주지 않는다.

디자인 판단:

운영 모니터라기보다 원시 로그 패널에 가깝다.

### 6.8 Project Comparison

근거:

- `src/web/src/components/ProjectComparison.tsx:22-60`
- 스펙: `docs/dashboard-panels.md` 8번 섹션

장점:

- 비교 테이블로 정보 구조는 명확하다.
- 숫자 정렬과 에러 컬러 강조는 합리적이다.

문제:

- 스펙에 있는 `Est.Cost`가 빠져 있다.
- 프로젝트 비교라기보다 집계 표에 가깝다.
- 한눈에 좋은 프로젝트와 문제 프로젝트를 구분하는 시각적 신호가 약하다.

디자인 판단:

테이블 자체는 깔끔하지만, 대시보드 맥락에서 인사이트 전달력은 낮다.

## 7. 스펙 대비 정합성 진단

### 7.1 스펙과 현재 구현이 잘 맞는 부분

- 전체 패널 구성이 스펙의 8개 패널 틀을 유지한다.
- 다크 대시보드 방향성과 미니멀 카드 스타일은 어느 정도 일치한다.
- Activity, Timeline, Files, Agents, Server, Projects라는 큰 축은 유지되고 있다.

### 7.2 스펙과 현재 구현이 어긋나는 부분

- `Activity Stream`: 필터, 상세 드릴다운, 에이전트 맥락 부재
- `Dev Server Monitor`: 요약 KPI 부재
- `Error & Alert Panel`: 에러 소스 통합 부재
- `Session Timeline`: 서사형 세션 조망 부재
- `File Heatmap`: 표현 방식 불일치
- `Agent Tracker`: 활동량 정보 부재
- `Project Comparison`: 비용 비교 누락

진단:

현재 구현은 "패널 골격 완성" 단계로 보는 것이 정확하다. 스펙에 정의된 대시보드 경험 전체를 구현한 상태는 아니다.

## 8. 사용자 경험 관점 진단

### 8.1 첫 화면 3초 인지

지금 화면은 첫 3초 안에 다음 질문에 답하기 어렵다.

- 지금 작업은 정상인가?
- 어떤 문제가 가장 시급한가?
- 어떤 에이전트나 파일이 병목인가?
- 방금 무슨 변화가 생겼는가?

이유:

- 강한 우선순위 영역이 없다.
- 핵심 지표가 헤더나 상단에서 요약되지 않는다.
- 각 패널이 "요약"보다 "원문 일부 나열"에 가깝다.

### 8.2 빈 상태 경험

`docs/dashboard-dark.png` 기준으로 빈 상태 화면은 세련되지만, 제품 데모나 온보딩 관점에서는 설명이 부족하다.

문제:

- 거의 모든 카드가 `No data`, `No agents`, `Clean`처럼 단문 상태만 보여준다.
- 대시보드가 무엇을 보여주는지 학습시키지 못한다.
- 데이터가 없는 순간의 브랜드 경험이 약하다.

### 8.3 운영형 가독성

운영 대시보드는 "미니멀"보다 "판단 속도"가 우선이다. 현재 UI는 미적 통일성은 있으나, 즉시 판단에 필요한 대비와 요약이 부족하다.

## 9. 우선 개선 과제

소스 수정 전 설계 기준으로 우선순위를 정리하면 다음과 같다.

### P0. 헤더를 상태 바화

- 브랜드명 중심 구조에서 운영 상태 중심 구조로 전환
- 세션 상태, 활성 에이전트 수, 에러 수, 최근 이벤트 시각, 서버 상태를 상단에 승격

### P1. 카드 계층 재정의

- 경고/핵심/보조 카드로 구분
- `Errors`는 지금보다 더 강한 시각적 우선순위를 가져야 함
- `Cost`는 큰 숫자보다 비교 문맥이 중요

### P2. 패널 명칭과 시각화 정합성 정리

- `File Heatmap` 유지 시 진짜 heatmap/treemap로 발전
- 그렇지 않으면 `Top Files` 같은 현실적인 명칭으로 조정
- `Agent Tracker`, `Session Timeline`도 동일한 기준 적용

### P3. 빈 상태 재설계

- 기능 소개형 empty state
- 예시 데이터 톤 또는 안내 카피 추가
- "현재는 비어 있지만 어떤 상황에서 값이 채워지는지" 설명 필요

### P4. 요약 대 원문 비율 재조정

- 현재는 원문 로그 비율이 높음
- 상단 요약, 하단 상세의 구조가 더 적절

## 10. 다음 단계 제안

문서화 이후 실제 설계 작업은 다음 순서가 적절하다.

1. 정보 위계 재설계
2. 로우파이 와이어프레임 작성
3. 패널별 역할 재정의
4. 시각 토큰 조정
5. 실제 UI 구현

## 11. 최종 판단

현재 대시보드는 구현 완성도보다 구조 가능성이 높은 상태다.  
즉, "방향은 맞지만 아직 대시보드답게 읽히지는 않는다"가 가장 정확한 진단이다.

다음 단계는 색이나 장식보다 먼저, 상단 상태 구조와 패널 역할을 다시 정의하는 와이어프레임 작업이어야 한다.
